## Phase 2（韌體強化）

### 2A — 按鍵重新配置（優先）

**目標：** 從 8 鍵藥物群組模式改為 PM 的 5 鍵語意配置

| 實體按鈕 | 新語意 | 原語意 |
|---------|--------|--------|
| 正下方大鍵（紅色）| 主鍵（短按記錄/確認、長按 ≥ 2s 開始/暫停任務）| BTN1 G0-Epi |
| 左側上鍵 | 模式選單：上一項 | BTN2 G0-Amio |
| 左側下鍵 | 模式選單：下一項 | BTN3 G0-D50 |
| 右側鍵 | 電源（短按螢幕喚醒/鎖定、長按 ≥ 2s 開機/關機）| BTN4 G0-TXA |
| 左上角鍵 | 錄音鍵（長按開始/停止錄音，儲存至 SD 卡）⚠️ 依賴 INMP441 + SD 卡硬體 | BTN5 選單 |

- 剩餘 3 顆（原 BTN6-8）暫時停用或備用
- 更新 GPIO 腳位映射與 debounce 設定
- 移除 G0/G1 藥物群組邏輯，改為模式（mode）概念
- 左上角錄音鍵功能先保留介面，實際錄音邏輯待 INMP441 麥克風硬體驗收後（Phase 1.5）再啟用

**驗收：** 5 顆按鈕各自語意正確觸發，長按短按分流正確（錄音鍵可先做 noop 佔位）

### 2B — 狀態機

**目標：** 實作 IDLE / RUNNING / PAUSE / END 四狀態

```
IDLE
  └─ 主鍵長按 ≥ 2s → RUNNING（建立任務、記錄開始時間、開始計時）
RUNNING
  ├─ 主鍵短按 < 1.5s → 記錄事件（依模式）
  ├─ 主鍵長按 ≥ 2s → PAUSE
  └─ 任務正常進行中
PAUSE
  ├─ 主鍵短按 → 回到 RUNNING
  └─ 主鍵長按 ≥ 2s → END（儲存資料、回到 IDLE）
END → IDLE（自動）
```

**實作要點：**
- 新增全域 `DeviceState` enum
- 現有 `handleButtonPress()` 加上狀態 guard
- IDLE 狀態下只有主鍵長按有效，其他按鍵 noop
- RUNNING 才啟動計時器與提醒循環
- PAUSE 暫停計時但保留資料在 RAM

**驗收：** OLED 顯示當前狀態，長按短按轉換正確

### 2C — 資料結構擴充

**目標：** 從 `EmsEvent` 擴充為接近 PM 的 `event_t`

```c
typedef struct {
    uint32_t event_id;       // 流水號，session 內自增
    uint64_t timestamp;      // epoch ms（RTC 或軟體對時）
    uint32_t elapsed_ms;     // 從任務開始的經過毫秒（保留）
    uint8_t  event_type;     // 事件類型（給藥/通氣/系統）
    uint8_t  source;         // 來源（0=按鍵, 1=系統）
    uint8_t  mode;           // 當下模式（0=給藥, 1=通氣, 2=自訂, 3=設定）
    uint8_t  sync_flag;      // 0=未同步, 1=已同步
    char     extra_data[16]; // 備註（縮短為 16 bytes 節省記憶體）
} EmsEvent;
```

- `device_id` 延後到主副機 BLE 架構實作時再加（避免現在沒有 ID 來源）
- `extra_data` 縮短為 16 bytes（PM 的 32 bytes 在容量 100 筆時佔 3.2KB，可接受，但先保守）
- MAX_EVENTS 從 30 提升到 100

**記憶體估算：** 100 筆 × ~36 bytes = 3.6KB，ESP32-S3 512KB SRAM 無壓力

**驗收：** BLE dump 輸出包含所有新欄位，JSON 格式正確

### 2D — 模式切換與給藥倒數提醒

**目標：** 實作 4 種模式選單 + 給藥模式的 240 秒倒數提醒主流程

**4 種模式（左側上下鍵瀏覽）：**

| 模式 | 提醒邏輯 |
|------|---------|
| 給藥模式（預設）| 240 秒（4 分鐘）倒數，到時強提醒（蜂鳴 + 震動馬達 + OLED 閃爍「請給藥」），主鍵短按確認重置倒數 |
| 通氣模式 | 顯示通氣計時，記錄通氣事件，無固定倒數循環 |
| 自訂模式 | 倒數時間可設定（Phase 2 先留介面，設定功能 Phase 3） |
| 設定模式 | 裝置設定（亮度、BLE、時間等，Phase 3 實作） |

**震動馬達硬體（新增採購）：**
- 型號：1027 扁平硬幣馬達（10×2.7mm，3V DC，~80mA）
- 驅動：S8050 NPN 電晶體 + 1N4148 二極體 + 1kΩ 基極電阻
- GPIO：從原 BTN6-8 空出的腳位中選一支（待確認）

**主流程（給藥模式）：**
```
RUNNING 進入 → 啟動 240 秒倒數
240 秒到 → 強提醒（蜂鳴 + OLED 閃爍「請給藥」）
主鍵短按 → 記錄給藥事件（timestamp + elapsed_ms）→ 重置 240 秒倒數
```

**注意：** 圖上的「6 秒節律」是副機（Assistant）收到主機給藥事件後的震動/蜂鳴持續時間，屬於 Phase 3C 主副機 BLE 架構的行為，此處不實作。

- 左側上下鍵在 IDLE 與 RUNNING 狀態下均可切換模式
- 模式切換時 OLED 顯示當前模式名稱
- 計時提醒只在 RUNNING 狀態下執行

**驗收：** 給藥模式 240 秒倒數正確觸發提醒，主鍵短按後重置計時，OLED 顯示剩餘秒數

- dump 命令輸出所有事件（格式升級）
- sync_flag 在 BLE 傳完後標記為 1

### 2F — 螢幕升級（⚠️ 等待採購決策）

**狀態：待確認採購型號，移至 Phase 3 決策後再動**

**2026-04-22 PM 新圖分析：**
PM 新版流程圖硬體規格寫的是 **2.4" TFT LCD 全彩液晶顯示**，不是 SH1106 OLED。
兩個選項：

| 選項 | 型號 | 介面 | Library | 解析度 |
|------|------|------|---------|--------|
| A（原計畫）| SH1106 1.3" OLED | I2C（現有接線）| U8g2 | 128×64 mono |
| B（PM 規格）| ILI9341 2.4" TFT | SPI | TFT_eSPI / Adafruit_ILI9341 | 240×320 彩色 |

**選項 B 的影響：**
- 接線需改 SPI（SCK/MOSI/CS/DC/RST），釋出 SDA/SCL
- 畫面可做全彩（紅色警報、綠色確認等）
- Library 換 TFT_eSPI，畫面重寫
- 邏輯層（狀態機、倒數、BLE）**完全不需改動**

**暫定：** 採購前先用現有 SSD1306 開發，邏輯穩定後一次換螢幕。
選定型號後再執行此任務。

## Phase 3（持久化 + App + 主副機）

### 3A — 持久儲存（LittleFS）

**目標：** 任務結束時將事件資料寫入 Flash，重開機不遺失

**實作方案：**
- 使用 ESP32 Arduino 內建 LittleFS
- 每次任務結束（END 狀態）→ 序列化為 JSON → 寫入 `/sessions/<task_id>.json`
- 開機時掃描 `/sessions/` 目錄，載入未同步的 session 列表
- BLE 傳輸完成後刪除或標記該 session 檔案

**檔案結構：**
```
/sessions/
  ├── 1713715200.json   ← 以 session 開始時間命名
  └── 1713718800.json
```

**驗收：** 任務進行中拔電重插，開機後 BLE dump 仍可取到上次任務資料

---

### 3B — DS3231 RTC 整合（原計畫）

- I2C 接線（與 OLED 共用 bus）
- 開機時從 RTC 讀取時間，不依賴 BLE 對時
- 若 BLE 連線則以 App 下發的 epoch 校正 RTC

---

### 3C — 主副機 BLE 架構（裝置對裝置）

**目標：** 兩台裝置一主一副，主機 Central 掃描並連接副機 Peripheral

**角色定義：**
- 主機：按下設定鍵進入 Central 模式，主動掃描副機
- 副機：預設 Peripheral 模式，廣播等待主機連接

**同步內容：**
- 時間同步（主機對時給副機）
- 指令同步（開始/結束任務）
- 事件上傳（副機事件傳至主機彙整）
- 給藥事件通知（主機記錄給藥 → 副機震動馬達 + 蜂鳴持續 6 秒）

**Service 架構（對應 PM 規格）：**
- Device Info Service：裝置 ID、電量、模式
- Sync Service：時間對時
- Command Service：任務控制指令
- Event Upload Service：事件資料批量傳輸

**注意：** 這會取代目前的 NUS 架構，是較大的 BLE 重構。在 3A/3B 完成後再動。

---

### 3D — 手機 App（React Native 建議）

依 PM 規格的 App 層實作：
- BLE 掃描 + 配對
- 接收事件資料（來自主機）
- 時間軸顯示
- SQLite 本地儲存
- 歷史紀錄列表
